# 무엇을 측정하나

**실제 과학 연구에서 나오는 연구 수준 코딩 능력**을 측정한다. 일반적인 코딩 벤치마크(웹 알고리즘 문제, 흔한 라이브러리 사용)와 달리, SciCode는 자연과학 연구자가 일상 워크플로에서 직접 쓰는 스크립트와 영향력 있는 논문의 방법을 코드로 재현하게 한다. 한 문제를 풀려면 세 가지가 동시에 필요하다 — **깊은 과학 배경지식**, 복잡한 문제를 더 작은 부분으로 쪼개는 **분석적 추론**, 그리고 각 부분을 올바르게 구현하고 통합하는 **코드 합성**. 즉 "과학적 사고를 코드로 옮기는" 능력 전반을 본다.

설계 동기는 명확하다. LM이 이미 평균적 인간을 여러 과제에서 능가하면서, 도전적이고 현실적인 평가를 만들기가 점점 어려워졌고 기존 벤치마크는 빠르게 포화됐다. SciCode는 16개 자연과학 세부 분야의 **과학자가 직접 큐레이션**해 이 문제를 정면으로 다룬다.

# 과제 형식 / 예시

- **80개 메인 문제**, 각각이 여러 개의 더 작고 쉬운 **서브문제로 분해**되어 총 **338개 서브문제**.
- 메인 문제당 서브문제는 중앙값 3개, 최대 15개. 데이터 분할은 개발용 15 메인(50 서브) / 테스트용 65 메인(288 서브).
- 모델은 서브문제마다 Python 함수를 하나씩 구현하고, 이를 통합해 메인 문제의 완전한 해답을 만든다.
- 각 문제·서브문제에는 입출력 형식을 명시한 **docstring**, 필요 시 과학자 주석가가 직접 쓴 **과학 배경지식**, 그리고 평가용 **gold 표준 해답**과 **다수의 테스트케이스**가 함께 제공된다.

예시(논문 Figure 1): "Haldane 모델로 육각 격자에서 파라미터를 훑어 Chern 수 배열을 생성하라"는 메인 문제가, (1) Haldane 해밀토니안 작성 (2) 해밀토니안으로 Chern 수 계산 (3) 2D 브릴루앙 영역을 격자로 이산화 같은 서브문제들로 쪼개진다. 정답 코드는 NumPy·SciPy·SymPy 등 널리 쓰이는 패키지만 사용하도록 작성된다.

# 채점 방식

- 지표는 **pass@1** — 생성한 코드가 테스트케이스를 통과하는지로 자동 채점.
- 테스트 스위트는 두 부분. (1) **수치 테스트(numerical tests)**: 입출력 쌍으로 ground truth와 출력이 일치하는지 확인. (2) **도메인 특화 테스트케이스(domain-specific test cases)**: 논문에 발표된 결과나 이론적 해석값을 재현하는지로 과학적 정확성까지 검증(예: 2D 정사각 Ising 모델의 상전이가 kT/J ≈ 2.269 부근에서 일어나는지).
- 채점 단위는 두 층위. **메인 문제**는 모든 서브문제 해답이 맞고 통합 해답까지 옳을 때만 성공으로 친다(매우 엄격). **서브문제**는 개별적으로 평가할 수도 있다.

# 점수 해석 (높음 / 낮음)

- **높음** = 과학 배경지식·문제 분해·정확한 코드 구현·부분 해답 통합을 모두 갖춰, 연구자가 실제로 쓸 만한 과학 코드를 만들어낸다.
- **낮음** = 서브문제 일부는 맞혀도, 메인 문제를 끝까지 통합해 풀어내지 못한다(현 LM 대부분이 여기에 해당).
- 서브문제 점수는 높은데 메인 점수가 급락하면 → **오류 누적과 통합 실패**가 병목임을 뜻한다.

# 난이도 · 포화 · 현 SOTA

논문(2024년 7월) 실험은 SciCode가 매우 어려운 벤치마크임을 보여준다. 가장 현실적인 평가 설정인 **표준 설정(standard setup)** — 배경지식 없이, 이전 서브문제에 모델이 직접 생성한 해답을 이어 쓰는 설정 — 에서:

- 메인 문제 pass@1: **Claude 3.5 Sonnet 4.6%**(최고), Deepseek-Coder-v2 3.1%, Claude 3 Opus·GPT-4o·GPT-4-Turbo·Gemini 1.5 Pro·Qwen2-72B 각 1.5%, Llama-3-70B·Mixtral-8x22B 0.0%.
- 서브문제 pass@1: 최고 모델들이 14~26% 수준(Claude 3.5 Sonnet 26.0%).
- **과학 배경지식을 제공해도** 최고 모델(Claude 3.5 Sonnet)의 메인 문제 정답률은 **12.3%**에 그친다 → 배경을 줘도 여전히 어렵다.

상위권조차 한 자릿수에 머물러, **포화와는 거리가 멀고 변별력 여유가 매우 크다.** (위 수치는 모두 논문 발표 시점 모델 기준. 최신 프런티어 모델 수치는 별도 리더보드를 참조.)

# 한계 · 주의

- **극단적 난이도로 인한 바닥 효과(floor effect)**: 발표 시점엔 거의 모든 모델이 메인 문제에서 0~5%에 몰려 약한 모델 간 변별이 어려웠다(서브문제 단위 평가로 일부 완화).
- **메인 문제 채점이 all-or-nothing**: 서브문제 하나만 틀려도 메인은 실패 → 부분적 역량이 점수에 잘 안 드러난다.
- **좁은 범위**: 자연과학 5개 도메인·Python 한정 → 일반 소프트웨어 엔지니어링·웹·시스템 코딩의 대리지표는 아니다.
- **분포 편중**: 메인 문제의 약 46%, 서브문제의 약 49%가 물리학(Figure 2). 도메인 균형이 고르지 않다.
- **평가 설정 민감성**: 배경 제공 여부, gold vs 생성 해답 이어쓰기 여부에 따라 점수가 크게 달라지므로 비교 시 동일 설정 확인 필수.

# 다른 벤치마크와의 관계

- 일반 코딩 벤치마크(HumanEval 등)는 흔한 알고리즘·라이브러리 위주라 빠르게 포화됐다. SciCode는 **연구 수준·과학 도메인**으로 천장을 다시 끌어올린 코딩 벤치마크.
- [[gpqa-diamond]] GPQA Diamond — 같은 자연과학 박사급이지만 **객관식 지식·추론**을 본다. SciCode는 그 지식을 **실행 가능한 코드로 구현**하는 데까지 요구한다는 점이 핵심 차이.
- 추론 축 벤치마크들이 "정답을 아는가"를 본다면, SciCode는 "그 정답을 작동하는 코드로 만들 수 있는가"를 본다.

# 레이더 기여 근거

- **coding (1.0)**: 본질이 연구 수준 과학 코드 생성 → 코딩 축에 단독 주 기여. 과학 지식·추론도 요구하지만 최종 채점이 코드 실행 통과(pass@1)이고 산출물이 코드이므로, 지식·추론 축으로 분산하지 않고 코딩 축에 온전히 귀속한다. (지식·추론 측면은 GPQA·HLE 등이 별도로 담당.)

# 한국 · 앨런 관점

이공계 연구자가 모델을 "코딩 보조 연구원"으로 쓸 때 얼마나 믿을 만한지를 가장 직접적으로 가늠하는 축이다. 수치 시뮬레이션·데이터 처리·논문 방법 재현 같은 실제 연구 코드 작성 신뢰도와 직결되며, 단순 챗봇 코드 스니펫이 아니라 과학적 정확성까지 검증한다는 점에서 연구·R&D 현장 활용도 평가에 특히 의미가 크다.
